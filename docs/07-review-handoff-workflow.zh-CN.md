# 审阅交接工作流

本文件说明怎样用当前 Reader 组织一次文件审阅交接。它面向工作流宿主、协调者和审阅参与者，不要求审阅参与者接触 Python、AI、源目录或源码。

对应英文文件：`07-review-handoff-workflow.en.md`

## 适用场景

当一轮审阅涉及几十份、甚至九十多份 Markdown、TXT 或 CSV 材料时，单纯发送压缩包或网盘目录会让审阅者自行理解文件结构、逐份打开并用聊天消息描述位置。

Reader 将选定范围打包为一个自包含的 `reader.html`，让审阅者在浏览器中连续阅读、搜索、筛选和批复，再用 `review.md` 把带坐标的意见交回中心。

## 两类角色

### 工作流宿主或协调者

中心侧负责：

- 选择本轮需要审阅的材料范围；
- 用配置文件生成一个或多个 `reader.html`；
- 按批次、角色、主题、轮次或展示范围分发审阅包；
- 收集审阅者返回的 `review.md`；
- 整理意见、维护源文件，并决定是否交给外部 AI 工作流继续处理；
- 生成下一轮需要审阅的材料。

### 审阅参与者

审阅参与者只负责：

- 用现代浏览器打开收到的 `reader.html`；
- 阅读、搜索和筛选自己要看的材料；
- 对具体内容块留下判断；
- 导出 `review.md` 并按约定交回。

审阅参与者不需要安装 Python、不需要项目依赖、不需要接触源目录，也不负责合并批复或修改源文件。

## 基本交接链路

```text
中心侧选择材料范围
  -> 生成 reader.html
  -> 分发给审阅参与者

审阅参与者阅读、判断、批复
  -> 导出 review.md
  -> 按约定交回中心

中心侧整理意见
  -> 维护源文件或交给外部 AI 工作流处理
  -> 重新生成下一轮 reader.html
```

## 用配置文件分包

一个配置文件可以定义一个审阅包的来源、匹配范围和输出位置。不同配置可以对应不同批次或展示范围：

```yaml
workspace_root: ..
source_dir: ./review-batches/batch-a
include:
  - "*.md"
  - "*.txt"
  - "*.csv"
output: ./output/batch-a.reader.html
```

也可以通过文件夹组织和匹配规则，让不同包分别承载不同主题、角色或轮次。配置不仅决定哪些文件进入包，也决定审阅者看到的材料组合和阅读顺序。

静态 HTML 一旦分发，接收者就能读取其中嵌入的内容。因此应在打包前确认范围，只把允许该审阅者持有副本的材料放进包内。

## 精确批复与复查

审阅者可以对段落、列表项、引用块、TXT 行和 CSV 行留下批复。每条导出的批复会保留：

- 文件路径；
- 内容块位置；
- 引用原文；
- 动作；
- 目标；
- 批复说明。

审阅板中的批复项可以跳回原文位置并短暂高亮。书签可以标记需要稍后复看的内容块。这让中心收到的不是“这里有问题”的聊天描述，而是可以重新定位的判断。

## 多人审阅的约定

几位审阅者或小团队可以直接使用多个审阅包并行审查。为保持回收清楚，建议在分发前约定：

- 每个包的批次或轮次；
- 每位审阅者负责的范围；
- 回执文件的命名方式；
- 交回时间；
- 由谁在中心侧整理和采纳意见。

例如，审阅者下载 `review.md` 后可按本轮约定重命名为：

```text
review_round-01_batch-A_reviewer-zhangsan.md
```

文件名中的审阅者、批次和轮次属于工作流约定，不是 Reader 自动写入的字段。

## 完成一轮后

审阅参与者应在完成前导出 `review.md`。浏览器会把批复暂存在本机 `localStorage`，但导出的回执才是可传递、可备份和可回收的交接物。

中心侧收到回执后，可以按文件和问题整理，再维护源文件或进入自己的后续处理流程。源文件始终不由浏览器直接写回。
